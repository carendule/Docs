# 设计多层继承架构模式


##### User:


请问在架构设计中，将一个类进行两次继承，分为子类和孙类，在父类中做主要业务对接，子类中做对外业务处理，孙类中做逻辑业务，这种设计模式或架构设计叫什么名字



##### Assistant:


这是一个非常具体且在传统企业级开发框架中常见的代码组织结构。严格来说，**在经典的 23 种设计模式（GoF）中，并没有一个专门的名字完全对应这种“三层继承”结构**。

这种做法更多是一种**架构惯例（Architectural Idiom）**或**分层超类（Layer Supertype）**的变体应用。

我们可以从以下几个维度来定义和分析这种设计：

### 1. 最核心的设计模式：模板方法模式 (Template Method Pattern)

从行为逻辑上看，这种三层结构最本质的内核是**模板方法模式**的扩展。

*   **父类/基类 (Root/Base)**：定义了顶层的算法骨架、通用工具方法或核心资源（如数据库连接、日志实例）。它可能定义了一些`final`方法来固定流程（“主要业务对接”）。
*   **子类/中间层 (Intermediate/Abstract)**：实现了部分抽象方法，专门处理“对外业务”（如参数校验、协议解析、异常统一处理）。它充当了一个适配层或钩子（Hook）的实现者。
*   **孙类/具体实现类 (Concrete)**：只关注核心的“逻辑业务”，填补剩下的抽象方法。

**命名建议**：如果你非要给它一个名字，可以称之为**“分层模板模式” (Layered Template Pattern)**。

### 2. 架构视角：层超类型 (Layer Supertype)

在 Martin Fowler 的《企业应用架构模式》(PoEAA) 中，提到了 **Layer Supertype**。
指在某一层的所有的类都有一个共同的超类（父类）。你描述的结构实际上是**多级层超类型**：
*   Level 1 (Parent): 整个系统的基类 (System Base)。
*   Level 2 (Child): 特定模块或接入层的基类 (Module/Interface Base)。
*   Level 3 (Grandchild): 具体的业务类。

### 3. 具体应用场景与命名

在不同的技术栈和场景下，这种结构有不同的俗称：

#### A. 框架基类设计 (Framework Base Class Design)
这是最常见的叫法。
*   **父类**: `BaseController` / `BaseService` (负责注入、上下文获取)。
*   **子类**: `OpenApiController` / `InternalController` (负责鉴权、限流、格式化响应)。
*   **孙类**: `OrderController` / `UserController` (负责具体业务)。

#### B. 钩子/适配器链 (Hook/Adapter Chain)
如果子类（中间层）的主要作用是将外部杂乱的输入转化为孙类可用的标准输入，那么子类充当了**适配器 (Adapter)** 的角色，同时利用继承链传递。

---

### 专家级分析：这种设计的优缺点

作为架构师，在采用这种设计时需要警惕以下问题：

#### 优点
1.  **代码复用（DRY原则）**：父类和子类解决了大量重复的“胶水代码”（连接代码、校验代码）。
2.  **标准化**：强制孙类只关注业务逻辑，规范了开发流程。
3.  **统一管控**：如果需要对所有对外接口增加“日志审计”，只需修改子类即可，孙类无感知。

#### 缺点（风险点）
1.  **脆弱基类问题 (Fragile Base Class Problem)**：
    *   这是继承最大的痛点。如果父类（主要业务对接）发生变化，可能会对子类和孙类造成不可预知的破坏。
    *   继承将“实现细节”暴露给了子类，导致高耦合。
2.  **层级过深 (Deep Hierarchy)**：
    *   通常建议继承深度不超过 3 层。这里已经占满了 3 层。如果未来孙类发现通过“对外业务处理”还不够，需要再细分，就会导致继承链过长，维护困难。
3.  **灵活性差**：
    *   假设有一个孙类，它只需要“逻辑业务”，但不需要“对外业务处理”（子类功能），在继承体系下它被迫继承了不需要的功能（违反接口隔离原则的思想）。

### 更好的替代方案：组合优于继承 (Composition Over Inheritance)

现代架构设计（如 Spring 生态、Go 语言设计哲学）更倾向于使用**组合**或**AOP（面向切面编程）**来解决这个问题，而不是多层继承：

1.  **AOP (面向切面)**：将“对外业务处理”（如日志、鉴权、异常捕获）做成切面（Aspect）或过滤器（Filter/Interceptor），而不是放在子类里。
2.  **策略模式 (Strategy Pattern)**：将业务逻辑封装成策略对象，注入到主流程中。
3.  **装饰器模式 (Decorator Pattern)**：如果需要对业务逻辑增加“对外处理”的功能，在外层套一个装饰器，而不是继承。

### 总结

这种设计没有唯一的专有名词，最准确的描述是：**基于模板方法模式的多级继承架构 (Multi-level Inheritance Architecture based on Template Method)**。

在代码评审中，如果看到这种设计，通常被称为**“基类膨胀”**或**“中间层抽象”**。只要层级控制在3层以内且职责清晰，在很多传统MVC框架中是完全可接受的工程实践。

##### Assistant:


在软件工程中，架构设计的专业名词非常多，通常分为**宏观架构（系统级）**和**微观架构（代码组织级）**。

基于你之前的问题（关于类继承的组织方式），我猜你可能更关注**“如何组织代码结构”**这一类。

以下是目前主流的架构设计专用名词，我将其分为三大类进行介绍：

### 一、 代码组织与内部架构（Micro-Architecture）
这里主要解决“如何把代码分层、解耦”的问题，是你上一个问题的高级替代方案。

#### 1. 分层架构 (Layered Architecture / N-Tier)
这是最经典、最通用的架构。
*   **结构**：通常分为 **表现层 (Presentation)** -> **业务逻辑层 (Business)** -> **数据访问层 (Persistence)** -> **数据库 (Database)**。
*   **特点**：严格的自上而下调用，下层不能调用上层。
*   **现状**：所有架构的始祖，但容易导致“大泥球”（代码耦合严重）。

#### 2. 六边形架构 (Hexagonal Architecture / Ports and Adapters)
*   **核心理念**：这也是为了解决你之前提到的“对外业务处理”依然耦合在继承链中的问题。它认为系统应该把“核心业务逻辑”保护在中间，外部的一切（HTTP请求、定时任务、数据库、消息队列）都通过**端口 (Ports)** 和 **适配器 (Adapters)** 接入。
*   **价值**：让业务逻辑完全不依赖具体的技术实现（如不依赖 Spring MVC，不依赖 MySQL 驱动）。

#### 3. 洋葱架构 (Onion Architecture)
*   **核心理念**：基于六边形架构的改进。由于其依赖方向像洋葱的一层层皮一样，**只能由外向内依赖**。
*   **特点**：最核心是Domain Model（领域模型），外围是Domain Service，再外围是Application Service，最外层是UI和Infrastructure。

#### 4. 整洁架构 (Clean Architecture)
*   **集大成者**：由 Robert C. Martin (Uncle Bob) 提出。它融合了六边形和洋葱架构的特点。
*   **关键图示**：著名的同心圆图。核心是 **Entities (实体)** 和 **Use Cases (用例)**。
*   **规则**：依赖原则（Dependency Rule），源代码的依赖关系只能指向内部圆心。

#### 5. 领域驱动设计 (DDD - Domain-Driven Design)
*   严格来说这是一种设计**方法论**，但它诞生了特定的架构模式。
*   **充血模型 (Rich Domain Model)**：反对只有 Get/Set 方法的贫血对象，主张把逻辑放回对象里（这与你之前的孙类只做逻辑业务有点像，但DDD是用组合而非继承）。
*   **界限上下文 (Bounded Context)**：将大系统拆分为一个个独立的业务领域。

---

### 二、 系统拓扑与分布式架构（Macro-Architecture）
这里解决“服务器之间如何通信、如何部署”的问题。

#### 1. 单体架构 (Monolithic Architecture)
*   所有功能打包在一个 WAR/JAR 包里运行。
*   **优点**：开发简单，测试方便。
*   **缺点**：牵一发而动全身，扩展性差。

#### 2. 微服务架构 (Microservices Architecture)
*   把单体拆分成几十个甚至上百个小服务（User Service, Order Service, Pay Service）。
*   **特点**：不仅是代码独立，**数据库也是独立的**（Database per Service）。
*   **BFF (Backend for Frontend)**：这是微服务中的一个专用名词，指专门为前端（Web、App、小程序）定制的聚合层后端。

#### 3. SOA (Service-Oriented Architecture, 面向服务架构)
*   微服务的前身。主要强调通过 **ESB (企业服务总线)** 来集成各个异构系统。现在在传统银行、电信行业还在用。

#### 4. 服务网格 (Service Mesh)
*   **代表技术**：Istio, Linkerd。
*   **理念**：把微服务中通信、熔断、限流等非业务逻辑剥离出来，放入一个独立的代理（Sidecar）进程中。被称为“第二代微服务”。

#### 5. 无服务器架构 (Serverless)
*   **代表技术**：AWS Lambda, 阿里云 FC。
*   **理念**：开发者只写函数（FaaS），不需要关心服务器、操作系统和容器。

---

### 三、 数据流动与高性能架构
这里解决“数据极其复杂、并发极高”时的设计问题。

#### 1. CQRS (Command Query Responsibility Segregation)
*   **命令查询职责分离**。
*   **理念**：把“读”和“写”的模型完全分开。
*   **场景**：比如电商详情页，读的请求是写的1000倍。写操作进入写库，然后异步同步到读库（如ElasticSearch/Redis）供查询。

#### 2. 事件驱动架构 (EDA - Event-Driven Architecture)
*   系统组件之间不直接调用，而是通过抛出“事件”（Message/Event）来协作。
*   **解耦神器**：A系统下订单 -> 发送“订单创建”事件 -> B系统收到事件扣库存 -> C系统收到事件发短信。

#### 3. 事件溯源 (Event Sourcing)
*   **理念**：不记录数据的“当前状态”，而是记录导致状态变化的“一系列事件”。
*   **例子**：银行账户余额不是一个数字，而是由“+100, -50, +20...”这些日志计算出来的。

#### 4. 微内核架构 (Microkernel / Plugin Architecture)
*   **插件化架构**。
*   **结构**：由一个核心系统（Core System）和插件模块（Plug-in Modules）组成。
*   **例子**：Eclipse、IDE、VS Code、Nginx 都是这种架构，核心很小，功能全靠插件。

---

### 总结与建议

如果你在做企业级业务开发，建议重点深入理解 **“整洁架构 (Clean Architecture)”** 和 **“DDD (领域驱动设计)”**。

这两个概念能帮你彻底解决你之前提到的“父类子类孙类如何分配职责”的困惑——它们会告诉你：**不要用继承来分配职责，要用分层和组合。**
